一个在网络上的家
Aug 12 2020

第一次建属于自己的个人博客网站，挺开心的，也算是在网络上有了一个属于自己的家。

这个网站是借助github pages和Hexo而建成的，在建站过程中看到一篇介绍特别详细的博客教程，如果有建立个人博客的想法的同学可以参考这篇博客。

建这个这个博客的时候还踩了点雷。一开始想用clover这个主题的，但是执行hexo s就一直报错，后来看到网上也有用clover却不显示的，就用vs code看了下，clover的源程序报错，然后改用了white。

关于white主题，我刚开始用的样子，主页是简洁的几行字，博客页则是现在white主题主页的样子。而在我更新white主题后，一开始有点不适应，但越看越喜欢，相较之前，博客的显示更直接。

然后就是封面和图床，封面的设置在网上找到了个说是photo，但是没用，就去问了原作者，属性是cover。图床的话，希望picgo稍微优化下，最后选择用的是百度到的路过图床，希望这个图床活得久一点。

github在国内总是随缘登陆，但是网易uu加速器可以免费加速github等学术资源，一开始使用的时候，uu加速器弹出的学术资源窗口的热门榜单第一是github，但几天后就没了，不过依然可以加速与github的连接。

在写博客的过程中，更多追求随性，博客的整体调整也是追求随性的一部分，在一些细节的记录上，例如引用带链接的文字段，既有写过前后加空格，也有前后不加空格的，反正选择看着舒服的就行，不必强制要求这些细节，不然会受到很多限制，甚至会因此产生些焦虑，担心是否有哪边没改好。此外，在写博客的过程中，也会存在对一件事情不同时间节点的描述、在一些表达中选择较好的一种方式、对于一些内容的留存选择等很多细节的处理。

关于hello-world这篇博客的封面在之前的描述中倒是不怎么提及，而这会（2022年5月初）也是在对博客中使用的图片梳理时，补充对这个封面的介绍：这个封面是大一任选课室内设计的一个作业，基础的房型图是老师提供的，那些更黑的笔迹则是我在手机上添加的室内设计整体方案，由于基础房型图的版权问题，这个封面依旧是归为不可商用的范围，最近也考虑过更换，但想来想去，还是继续用这张图做这篇博客的封面。

域名更改

2020年10月底，个人博客域名改为islandfsj.top，此前使用的02-16.com则用于零二之一六的官网域名。

博客的整体调整

2022年3月中旬开始对博客进行一些整体调整，先在hello-world里制定些调整后的规定，然后再逐篇修改，由于调整量比较大，所以从决定调整到开始调整间隔了几天。

写的东西，有的是为了记录，会很详细甚至繁杂，有的是为了向别人传递些信息，则会较为简练，有的也会兼具这两个目的，则会介于两者之间，想清楚写的东西是哪种类型对于写博客这件事来说很重要，同时对于其他的一些事情来说也很重要，因此从这个角度整理了下对于写博客这件事的一些整体调整：

1、刚开始写博客时（2020年8月）-2021年底：纸上、手机便签（记录）=>博客（传递信息，虽然整合但不稳定）

2、2021年底-2022年3月：纸上、手机便签（记录）=>博客（记录+传递信息）=>其他形式（更为整合和稳定地传递信息）

3、2022年3月以后：纸上、手机便签（记录）=>博客（传递信息，虽然整合但不稳定）=>其他形式（更为整合和稳定地传递信息）

在1和2中，对于心理活动的记录越来越多，特别是在2中，博客出现了记录的目的。1的时候，还只是从没想过记录一些心理活动，到开始想是否需要记录一些心理活动；2的时候，则是逐渐转变成会去记录一些心理活动，而后又会记录过多，甚至很多时间用在记录上，就又会开始克制。

例如对于如何记录博客的想法，某个地方为什么这么写；还有一些记录时用词的想法记录，会写类似于原本想的用词是什么，然后什么词更加合适；还有之前打错字，后来什么时候又更改了等等很多心理活动的记录。

其中，最有意思的是对博客改动的记录：在博客有修改的地方加括号，写着修改时间；文末分割线及标注时间（主要是在hello-world中，常用分割线样式：<<<—手动分割线—>>>）；博客记录分段（在2021年12月底写创业的逐步成型等博客时想出，并后续完善了使用的规定，一般格式为XXXX年XX月XX日 | 修改了哪些内容）。大致是这几类，可能有遗漏。

关于定稿，在2中创业的逐步成型博客就有过定稿的想法，但是由于是博客的形式，是一个依然可以再次修改的形式，即使在2022年2月底想到将本地文件改为只读（由于文件本身的调整不可避免，所以那时在除hello-world外的每篇博客里都增加博客记录分段，hello-world一直使用文末分割线），依旧难以在博客上完成定稿，还是需要其他形式才能完成定稿，例如写成图书，所以2022年3月中旬的时候也取消了将本地文件改为只读的方法。

那么在3中，博客则会回归只有传递信息的目的，那些细节的记录只需放在纸上和手机便签，当然也需要一定的克制从而保证正常的生活，博客里将追求内容的整合，更加简洁且突出重点，并追求还原事情发展的过程，但不再记录博客记录时的修改时间、内容更改等内容，删除博客记录分段等记录博客改动的各类形式。

形象地说，就是1的时候是在写作文，但又有点写日记的成分，对之前的内容做修改的时候，有时会想要不要留下修改的信息；2的时候是在博客上记日记，也会对之前的内容做修改，大多数时候也会留下修改的信息；3的时候，则是回归在博客上写作文，没有写日记的成分，也会对之前的内容做修改，但不会留下修改的信息，最多只有个别用于说明博客调整的信息。

在2中，在不方便写博客或是克制自己写博客的时候，特别是日记相关的部分，也会克制自己在纸上记日记以避免占用过多时间，在2最后的几天里，先是想到在手机便签里记录关键词，然后再在纸上或是博客上扩写（当时连着几天把“扩写”写成了“拓写”），决定改成3的当天则是想出每天固定时间写日记和博客，大概每晚9点、10点之后，其余时间用手机便签记录关键词，但还是太占用时间，且觉得用博客来做内容整合的效果会好于记录。

更换电脑时，转移本地博客方法

2022年1月第，换了新电脑（联想拯救者R9000P2021），将本地博客挪到新电脑上，以下几行则是之前尝试出的一个比较快速的转移本地博客方法：

①下载git、nodejs

②安装Hexo（npm install -g hexo-cli）

③创建本地博客（hexo init myBlog||cd myBlog||npm install）

④将该新建本地博客theme文件夹、source文件夹、_config文件替换为旧电脑本地博客的文件（可执行hexo s测试本地博客是否正常运行）

⑤配置SSH KEY

⑥在本地博客文件下安装部署插件（npm install hexo-deployer-git –save）并完成上传（hexo g -d）

图片的版权

由于个人博客不涉及任何商业用途（网站备案时有明确定位），一些图片放在个人博客里没什么问题，但考虑到一些图片有商用的可能性，所以仍然会对个人博客里从商用角度来说涉及版权的图片做修改，但是例如logo本身就使用字体等无法修改为不涉及版权问题的图片除外，在个人博客编辑页面的图片名称里会给不可商用的标明这一限制（具体的图片更改记录就不再列出了）。

图片修改时间：2022年4月底-2022年5月初。

hexo本地图片

将图片放入themes/（主题）/source文件夹里，那么在posts文件夹的博客文章里调用图片的路径为/themes/（主题）/source/（图片名称.图片后缀）。这样在vs code预览里没问题，但是上传后发现不显示图片，也尝试了搜索到的一些方法，也都没效果。

考虑本地图片是因为创业logo那张图没法上传常用的路过图床，本地图片也试不起来，就也试了下qq相册，也是显示不了，所以重新找了几个图床，但很多速度都不行，最后用的是速度ok的聚合图床。

几天后（2022年5月初）又尝试了本地图片，参考Hexo 引用本地图片以及引用本地任意位置图片的一点思路这篇博客，开启本地静态资源的post_asset_folder: true和给引用的本地图片添加绝对路径的hexo-asset-image与之前一致，然后在这篇博客中提到了兼容懒加载的hqweay/hexo-asset-image。

在弄好这三步操作后，在source/_posts下新建文件夹img，并放入番薯夹logo配图.jpg，调用的图片链接则是img/番薯夹logo配图.jpg，VS Code的预览中会显示图片，但是上传后依旧不显示。

按F12查看网页的html代码，data-src中的图片指向已经是绝对路径了。

再查看报错的指令，发现对于该图片的请求路径path并不是直接指向img，而是指向该与博客md文件所处的文件夹中，github代码库中也没有找到img文件夹。

所以还是得给每个博客建立同名文件夹，重新上传后，在github代码库中看到了islandfsj.github.io/2022/04/24/个人自媒体/番薯夹logo配图.jpg，但是依然存在图片上传至github后无法显示图片，报错内容则是无法加载图片。

值得一提的是，我之前在public文件夹下放入logo.png，用于网页，在github代码库中则是islandfsj.github.io/logo.png，在white/_config.yml里则是icon: logo.png，
